# 无后端 MVP 商业化开发计划

> 生成日期: 2026-06-20
> 适用对象: 创始团队内部决策、老板阶段汇报、研发任务拆解、种子客户试点准备
> 核心结论: MVP 第一版不做用户登录、不做 token 计费、不做代理模型、不要求本地轻量服务常驻。产品定位为“本地桌面 AI Agent + 用户自带模型 Key / 本地模型 + 本地 License 授权”。

## 1. 管理层摘要

当前客户端已经从“默认依赖有道/网易云服务”改到“默认本地 / BYOK 模式”的骨架状态。下一阶段不应继续扩展云能力，而应把产品收口成一个可以交付给种子客户的无后端 MVP。

MVP 的商业目标是用最小成本验证客户是否愿意为桌面 AI Agent 工具付费。第一版不承担模型 token 成本，不建设账号系统，不做代理转发，不做复杂订阅。客户使用自己的 API Key、企业模型网关、本地 Ollama / LM Studio 等模型能力；我们销售软件授权、配置服务、行业模板和后续维护。

推荐老板决策口径:

| 决策项 | 推荐选择 | 原因 |
| --- | --- | --- |
| 第一版商业模式 | 本地桌面软件授权 + BYOK | 成本低、合规轻、交付快 |
| 是否需要 `venusAI_Server` | MVP 不需要 | 不做登录、计费、模型代理时没有必要强依赖服务端 |
| 是否做用户登录 | 不做 | 两人团队暂不承担账号、安全、短信、客服和隐私合规成本 |
| 是否代理模型 | 不做 | 避免 token 成本、限流、滥用、数据合规和退款风险 |
| 授权控制 | 本地签名 License | 可离线交付，技术实现简单，足够支撑早期销售 |
| 第一批客户形态 | 种子客户 / 项目制 / 年授权 | 比 ToC 订阅更适合当前团队规模 |

## 2. 当前状态与 MVP 差距

### 2.1 已具备的基础

基于当前仓库和文档，客户端已具备以下基础:

1. 默认配置已经切到 `local-byok`，服务端模型默认禁用。
2. 新增统一 `RemoteServicesConfig`，主进程和渲染进程端点从统一配置解析。
3. 本地/BYOK 模式下已经短路登录、quota、服务端模型、媒体生成、实时 ASR、HTML 分享等云能力。
4. OpenClaw 配置同步在服务端模型为空时不再注入 `lobsterai-server` provider。
5. 用户可见品牌归属已经完成一轮 TopVan 替换。
6. 已有定向测试覆盖默认端点、OpenClaw provider 注入边界、日志导出等关键行为。
7. `E:\code\venusAI_Server` 已有兼容服务雏形，但 MVP 不把它作为必需运行组件。

### 2.2 距离无后端 MVP 的主要缺口

| 缺口 | 影响 | 优先级 |
| --- | --- | --- |
| 默认仍有 `127.0.0.1:8787` 文档/catalog 占位 URL | 未启动服务时部分入口失败 | P0 |
| 新用户模型配置路径不够产品化 | 客户不知道如何开始使用 | P0 |
| 本地 License 授权体系缺失 | 无法商业交付和控制试用期 | P0 |
| 打包安装包和干净环境验收未完成 | 不能交给客户安装 | P0 |
| OpenClaw + BYOK 真实端到端验证不足 | 代码测试通过不等于产品可用 | P0 |
| 网易/有道/POPO/联系邮箱等残留未完全清理 | 商业品牌和合规风险 | P1 |
| 诊断、日志、常见错误文档不足 | 售后成本不可控 | P1 |
| 第三方依赖授权清单未形成 | 对外销售存在法律不确定性 | P1 |

## 3. MVP 产品边界

### 3.1 MVP 必做

MVP 必须让客户在没有任何 TopVan 后端服务的情况下完成核心流程:

1. 安装客户端。
2. 导入或激活本地 License。
3. 配置模型 Provider，例如 DeepSeek、OpenAI-compatible、Ollama、LM Studio。
4. 测试模型连接。
5. 创建 Cowork 会话并完成一次真实任务。
6. 查看和导出日志。
7. 在云功能禁用时得到明确、友好的提示。

### 3.2 MVP 不做

第一版明确不做以下能力:

| 不做项 | 处理方式 |
| --- | --- |
| 用户注册和登录 | 使用本地匿名用户状态 |
| token 计费和额度余额 | 不展示充值、邀请、套餐余额等入口 |
| 自有云模型代理 | 用户直接连接自己的模型 Provider |
| 实时 ASR | 默认禁用 |
| 媒体生成 | 默认禁用 |
| HTML 公开分享 | 默认禁用 |
| 在线 Skill / Kit / MCP 市场 | 使用内置空列表、本地静态资源，或先隐藏入口 |
| 自动在线更新 | 第一版可关闭，改为手动下载安装包 |
| 企业 SSO / 多租户后台 | 第二阶段再做 |

### 3.3 `venusAI_Server` 的定位

`venusAI_Server` 不进入 MVP 必需链路。它保留为第二阶段选项:

1. 在线 License 激活和续期。
2. 自有更新 catalog。
3. 企业组织、seat、管理员后台。
4. 可选统一模型代理。
5. 私有化客户的轻量控制面。

MVP 当前目标是验证桌面产品价值，不是建设云平台。

## 4. 技术开发计划

### 阶段 0: 范围冻结与基线验证

目标: 确认当前改造能作为无后端 MVP 的基础。

任务:

1. 冻结 MVP 范围: 本地 BYOK、无登录、无服务端模型、无模型代理、本地 License。
2. 运行现有测试: remote services、endpoints、config、auth、本地云能力门控、OpenClaw config sync。
3. 搜索运行时代码中的 `youdao.com`、`netease.com`、`127.net`、`api-overmind`、`lobsterai-server`，区分可控官方模式常量和实际默认路径。
4. 输出一页状态报告: 已完成、待收口、风险、是否进入开发阶段。

验收:

1. 默认配置不落到官方服务。
2. 服务端模型禁用时 OpenClaw 不注入 `lobsterai-server`。
3. 不启动 `venusAI_Server` 时，客户端核心功能不应崩溃。

### 阶段 1: 无后端客户端收口

目标: 客户端不依赖 `127.0.0.1:8787` 才能优雅使用。

当前进度:

1. 本地/BYOK 默认远程端点已开始切换为内部占位协议 `local-byok://...`，避免默认请求未启动的本地服务。
2. Skill / Kit / MCP 市场将在本地/BYOK 模式下返回空列表，不再发起网络请求。
3. 本地/BYOK 模式下更新检查已短路为“无可用更新”，不会请求占位 URL。
4. IM 配置指南入口已先行隐藏；NIM 扫码登录默认禁用，不再请求本地兼容服务。
5. MVP 暂不内置 Computer Use runtime；相关下载地址已改为内部占位，安装路径默认返回未包含提示。
6. 本地/BYOK 账号菜单已隐藏云账号入口，不再展示剩余额度、充值、邀请或退出登录。
7. Portal 定价/订阅外链已加本地占位保护，不会打开 `local-byok://...`。
8. Youdao 智云 Provider 已从 MVP 可见模型 Provider 列表隐藏，底层定义保留用于兼容旧配置。
9. NIM、NetEase Bee、POPO 已从 MVP 可见 IM 平台列表隐藏，底层 channel 解析保留用于兼容旧配置。

任务:

1. 将本地/BYOK 默认模式从“默认请求本地服务”改为“默认无后端”。
2. 文档、服务条款、关于、社群入口改为:
   - 打包内置 Markdown/HTML 页面；或
   - 指向公司官网静态页面；或
   - 在 MVP 中隐藏非必要入口。
3. 更新检查默认关闭，或显示“当前版本请手动下载更新”。
4. Skill Store、Kit Store、MCP Marketplace 在无后端模式下返回空列表，不弹错误。
5. Computer Use runtime、Kit 下载、图标等远程资源在 MVP 中:
   - 若不作为主卖点，默认隐藏或禁用；
   - 若作为主卖点，必须随安装包内置或提供明确本地路径。
6. 云功能入口统一显示“当前版本未启用”，不能出现原厂服务名或请求失败堆栈。

验收:

1. 不启动任何本地服务，打开客户端无阻塞错误。
2. 断网状态下仍能打开主界面、设置页、模型配置页、日志导出页。
3. 所有 marketplace/catalog/update 请求失败都不影响核心 Cowork 流程。

### 阶段 2: BYOK 首次使用体验

目标: 新客户能在 5 分钟内配置模型并跑通 Cowork。

任务:

1. 在首次启动或模型为空时展示模型配置引导。
2. 提供 Provider 预设:
   - DeepSeek
   - OpenAI-compatible
   - Anthropic
   - Qwen / DashScope
   - Moonshot / Kimi
   - Ollama
   - LM Studio
3. 每个 Provider 提供默认 base URL、模型示例、帮助链接。
4. 增加连接测试:
   - API Key 缺失
   - base URL 不可达
   - 模型名错误
   - 代理/网络失败
   - 上游鉴权失败
5. 敏感 API Key 使用现有安全存储能力或至少避免明文展示。
6. 未配置模型时，发送 Cowork 任务前显示明确提示，而不是让 OpenClaw 启动后失败。

验收:

1. 新用户配置 DeepSeek 后能完成一次 Cowork 会话。
2. 新用户配置 Ollama 后能完成一次本地模型 Cowork 会话。
3. 故意填错 API Key 时，错误提示能指向具体修复方向。

### 阶段 3: 本地 License 授权

目标: 不做登录也能完成商业授权控制。

推荐方案: 签名 License 文件 / License Key。

架构:

1. 公司内部保留私钥。
2. 客户端内置公钥。
3. License 内容明文可读，但必须带数字签名。
4. 客户端验签、检查过期时间、检查 feature、可选检查机器码。

License 字段建议:

```json
{
  "product": "topvan-ai-agent",
  "customer": "customer name",
  "plan": "trial",
  "seats": 1,
  "features": ["cowork", "byok", "local-model", "skills"],
  "machineBinding": "optional-machine-code",
  "issuedAt": "2026-06-20",
  "expiresAt": "2026-07-20"
}
```

开发任务:

1. 新增共享 License 类型和常量。
2. 新增主进程 License 校验模块:
   - 读取 license 文件或 SQLite kv。
   - 使用 Ed25519 或 RSA-PSS 公钥验签。
   - 返回有效、过期、签名错误、机器不匹配、feature 不足等状态。
3. 新增机器码生成:
   - 使用本机稳定信息生成 hash。
   - 不显示原始硬件信息。
   - 允许客户复制机器码给销售/实施人员。
4. 新增内部 CLI:
   - 输入客户名、有效期、feature、机器码。
   - 输出 license 文件。
   - 私钥只放 CLI 环境，不进入客户端。
5. 设置页新增 License 面板:
   - 当前授权状态。
   - 到期时间。
   - 客户名称。
   - 机器码复制。
   - 导入 license 文件。
6. 功能门控:
   - 无 license 可允许 7-14 天试用，或要求导入试用 license。
   - 过期后禁止新建 Cowork 任务。
   - 保留历史数据和设置，避免客户数据被锁死。

验收:

1. 篡改 license 过期时间后验签失败。
2. 过期 license 禁止新建 Cowork。
3. 机器绑定 license 在另一台机器上无效。
4. 未授权状态不会清除用户数据。
5. 内部 CLI 可生成试用版和正式版 license。

### 阶段 4: 品牌、合规与依赖清理

目标: 对外交付时不出现原厂归属误导和明显授权风险。

任务:

1. 清理用户可见品牌:
   - 关于页联系邮箱。
   - 窗口标题。
   - 安装包名称。
   - 桌面图标、托盘图标。
   - 默认 Agent 身份。
   - README、服务协议、隐私说明。
2. 梳理第三方依赖授权:
   - 主仓库 MIT。
   - OpenClaw 授权。
   - OpenClaw 插件授权。
   - Electron、React、Vite、better-sqlite3 等 npm 包授权。
   - Skills 目录下 LICENSE。
   - 图标、文档、图片、字体资源。
3. 对网易/有道/POPO/NetEase IM 相关能力做产品决策:
   - 不卖的能力默认隐藏或禁用。
   - 保留的能力必须标明第三方服务依赖和客户自行配置。
4. 新增第三方声明文件:
   - `THIRD_PARTY_NOTICES.md` 或安装包内 notices 页面。

验收:

1. 普通用户界面不出现原厂出品/维护/联系邮箱。
2. 安装包元信息归属 TopVan。
3. 授权清单可供老板和法务/外部律师复核。

### 阶段 5: 安装包、诊断与客户试点

目标: 形成可以发给种子客户的交付包。

任务:

1. 打包 Windows 安装包。
2. 在干净 Windows 环境验收:
   - 无开发依赖。
   - 无 `venusAI_Server`。
   - 可导入 license。
   - 可配置模型。
   - 可跑通 Cowork。
3. 增强诊断:
   - 一键导出日志。
   - OpenClaw runtime 状态。
   - 模型连接测试结果。
   - 系统代理状态。
   - Node/Python/runtime 资源状态。
4. 编写客户文档:
   - 安装指南。
   - License 导入指南。
   - DeepSeek 配置指南。
   - Ollama 配置指南。
   - 常见错误排查。
   - 数据隐私说明。
5. 选择 1-3 个种子客户试点，记录:
   - 安装耗时。
   - 首次配置耗时。
   - 首次成功任务耗时。
   - 常见报错。
   - 客户愿意付费的场景。

验收:

1. 非开发人员按文档能完成安装和首次任务。
2. 客户环境出问题时，研发可以根据日志定位到模型、网络、授权、OpenClaw 或客户端问题。
3. 试点客户至少完成 3 个真实业务任务。

## 5. 里程碑排期

以两人团队为前提，建议按 6 周完成种子客户 MVP。

| 周期 | 目标 | 交付物 |
| --- | --- | --- |
| 第 1 周 | 范围冻结、无后端收口设计 | MVP 范围确认、缺口清单、任务拆分 |
| 第 2 周 | 客户端无后端收口 | 无 `127.0.0.1:8787` 依赖的客户端默认路径 |
| 第 3 周 | BYOK 引导和连接测试 | 模型配置向导、DeepSeek/Ollama 跑通 |
| 第 4 周 | 本地 License | License 校验、导入 UI、内部生成 CLI |
| 第 5 周 | 打包、品牌、诊断 | Windows 安装包、日志导出、品牌清理 |
| 第 6 周 | 种子客户试点 | 试点包、安装文档、问题清单、老板汇报 |

如果只做内部演示版，可压缩为 2 周:

1. 第 1 周完成无后端收口和 BYOK 跑通。
2. 第 2 周完成 License 简版、打包和演示脚本。

## 6. 汇报机制

建议每周给老板一次 30 分钟同步，使用固定格式，避免陷入技术细节。

### 每周汇报模板

1. 本周结论: 继续推进 / 需要决策 / 风险升高。
2. 已完成: 用客户价值表达，不只说改了哪些文件。
3. 演示证据: 截图、安装包、日志、一次真实任务结果。
4. 当前风险: 合规、成本、稳定性、交付、售后。
5. 下周目标: 只列 3-5 个可验收事项。
6. 需要老板决策: 价格、客户、品牌、合同、是否砍功能。

### 老板应关注的指标

| 指标 | 目标 |
| --- | --- |
| 首次安装成功率 | 种子客户环境可安装 |
| 首次模型配置耗时 | 5-10 分钟内 |
| 首次 Cowork 成功率 | 至少 80% 试点环境成功 |
| 无后端运行能力 | 不启动任何服务也可核心可用 |
| 单客户支持耗时 | 首次交付问题可在 1 小时内定位 |
| 模型成本 | 公司为 0，由客户 BYOK 承担 |

## 7. 风险清单与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| OpenClaw runtime 在客户机启动失败 | 核心功能不可用 | 增加状态检测、日志导出、安装包内置依赖检查 |
| 用户不会配置模型 | 试用失败 | 做首次引导、Provider 预设、连接测试和文档 |
| License 被复制传播 | 收入损失 | 签名 license + 可选机器绑定，早期足够 |
| Electron 客户端被逆向 | 授权被绕过 | 私钥不进客户端，避免追求不可实现的绝对防破解 |
| 残留原厂品牌或服务 | 合规和客户信任风险 | 发版前做运行时代码和 UI 全量扫描 |
| 第三方依赖授权不清 | 商业销售风险 | 建立授权清单，必要时外部律师复核 |
| 售后问题过多 | 两人团队被拖垮 | 先限制支持范围，只支持指定模型和 Windows 版本 |
| 客户需要账号/团队管理 | MVP 功能不足 | 记录需求，第二阶段用 license server 或轻量控制面解决 |

## 8. 第二阶段路线

MVP 验证客户愿意付费后，再考虑建设轻量服务端。

第二阶段可以做:

1. 在线 license 激活和续期。
2. 企业 seat 管理。
3. 自有更新 catalog。
4. 文档站和下载页。
5. Skill / Kit / MCP 静态市场。
6. 可选 `venusAI_Server` 生产化。
7. 可选模型代理，但必须先有:
   - 限流。
   - 成本统计。
   - 组织 quota。
   - 审计日志。
   - 数据隐私条款。
   - 上游模型供应商合规确认。

第二阶段的触发条件:

1. 至少 1-3 个种子客户完成真实试用。
2. 至少 1 个客户愿意付费或签试点协议。
3. 客户明确需要团队管理、在线更新或统一授权。
4. 当前无后端模式的售后问题已经可控。

## 9. 最终验收清单

MVP 发布前必须全部通过:
