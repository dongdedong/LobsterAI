# LobsterAI 核心技术全景分析

> 版本: 2026.6.18 | 分析日期: 2026-06-19

---

## 一、项目定位：这是什么？

**LobsterAI 是一款企业级 AI 协同桌面应用**，由 TopVan 出品，基于 Electron + React 构建。它不是一个简单的聊天 UI 壳子——它是一个**完整的 AI Agent 运行平台**，能够：

1. **在本地运行 AI Agent**（通过内嵌的 OpenClaw 引擎），让 Agent 直接操作用户的文件系统、终端和工具
2. **连接 10+ 即时通讯平台**（微信、企业微信、钉钉、飞书、Telegram、Discord、QQ、POPO 等），让 Agent 7×24 小时在线服务
3. **执行定时任务**（Cron Job），定期运行 Agent 并将结果推送到 IM 渠道
4. **渲染丰富的交互产物**（HTML、SVG、Mermaid、React 组件、Office 文档等）

简单来说：**它让 AI 不只是回答问题，而是能「做事」并「主动汇报」。**

---

## 二、解决的核心问题

| 问题 | LobsterAI 的解法 |
|------|-----------------|
| AI 只能在浏览器中聊天，无法操作本地环境 | 内嵌 OpenClaw Gateway，Agent 可直接读写文件、执行终端命令、操作浏览器 |
| 企业需要将 AI 连接到自有 IM 系统 | 统一 IM 网关层，一套 Agent 同时服务微信/钉钉/飞书/Telegram/Discord 等 |
| AI 需要定时执行任务（日报、巡检、提醒） | 内置 Cron 调度引擎 + IM 投递，支持自然语言创建定时任务 |
| 不同 LLM 提供商的 API 格式碎片化 | 多 Provider 适配层，统一支持 Anthropic / OpenAI / Gemini / 国内大模型 |
| AI 执行操作需要用户审批 | 权限请求机制（Permission Modal），每个工具调用都可以让用户确认或拒绝 |
| Agent 输出需要富格式渲染 | Artifacts 系统，支持 HTML/SVG/Mermaid/React/Excel/PPT/PDF 实时预览 |

## 四、架构核心设计理念

### 4.1 三层分离原则

```
┌───────────────────────────────────┐
│  GUI (渲染进程)                     │  ← 只负责「展示」和「交互」
│  React + Redux + Tailwind          │
├───────────────────────────────────┤
│  Cowork (主进程业务层)               │  ← 负责「业务编排」和「状态管理」
│  CoworkEngineRouter + CoworkStore  │
├───────────────────────────────────┤
│  Engine (执行引擎)                  │  ← 负责「执行 Agent 任务」
│  OpenClaw Gateway (子进程)         │
└───────────────────────────────────┘
```

**关键洞察**：Cowork Agent 执行链路中，渲染进程不直接发起 LLM 网络请求，也不直接操作文件系统——所有操作都通过 IPC 路由到主进程，再由 OpenClaw 引擎执行。渲染进程的 `services/api.ts` 虽然封装了多 Provider 调用逻辑，但其网络请求也通过 `api:stream` IPC 代理到主进程发出。

### 4.2 安全沙箱模型

- **context isolation**: 渲染进程完全隔离，无法访问 Node.js
- **contextBridge**: 仅暴露白名单 API（`window.electron.*`）
- **sandbox**: 启用 Electron 沙箱
- **工具审批**: Agent 的每一个工具调用都需要用户在 `CoworkPermissionModal` 中审批

### 4.3 字符串常量集中管理

所有 IPC 通道名、状态码、模式选择器都通过 `as const` 对象集中定义（参考 `src/scheduledTask/constants.ts`），避免魔法字符串散落导致的隐性 bug。

## 五、关键业务流程

### 5.1 一次 Cowork 会话的完整生命周期

```
用户输入 Prompt
    │
    ▼
CoworkView.handleStartSession()
    │
    ├─ 检查 OpenClaw 引擎状态 (phase === 'running'?)
    ├─ 检查 API 配置 (有 Provider + API Key?)
    ├─ 创建临时会话 (temp-*) 立即展示用户消息
    │
    ▼
IPC: cowork:session:start ──→ main.ts
    │
    ├─ 确保 OpenClaw Gateway 已启动
    ├─ CoworkStore.createSession() → SQLite
    ├─ CoworkEngineRouter.startSession()
    │     └─ OpenClawRuntimeAdapter.startSession()
    │           └─ WebSocket → OpenClaw Gateway
    │
    ▼
OpenClaw Gateway 执行 Agent 循环
    │
    ├─ 调用 LLM API (流式)
    ├─ 解析工具调用 → 发回 permissionRequest
    │     └─ GUI 弹出 CoworkPermissionModal
    │           └─ 用户 approve/deny → 返回结果
    ├─ 执行工具 (读写文件/终端/浏览器)
    ├─ 继续 LLM 循环直到完成
    │
    ▼
流式事件回传
    ├─ message → 新消息写入 SQLite + 推送 GUI
    ├─ messageUpdate → 流式内容增量更新
    ├─ complete → 会话结束，清理状态
    └─ error → 错误处理
```

### 5.2 IM 消息处理流程

```
IM 平台来消息 (钉钉/飞书/Telegram/...)
    │
    ▼
OpenClaw Channel Adapter 接收
    │
    ▼
IMGatewayManager.messageHandler()
    │
    ├─ 检测是否为定时任务请求 (正则 + LLM 提取)
    │     └─ 是 → 创建 CronJob + 返回确认消息
    │
    ├─ 交给 IMCoworkHandler.processMessage()
    │     └─ 创建/复用 Cowork 会话
    │     └─ Agent 执行完毕 → 回复消息
    │
    └─ 通过平台 API 发送回复
```

### 5.3 Artifacts 产物渲染流程

```
Agent 输出代码块 (```html / ```mermaid / ```react / ...)
    │
    ▼
ArtifactParser 检测 (显式标记 + 启发式推断)
    │
    ▼
ArtifactPanel 右侧面板渲染
    │
    ├─ HTML → sandboxed iframe (无 allow-same-origin)
    ├─ SVG → DOMPurify 消毒后渲染 + 缩放控制
    ├─ Mermaid → Mermaid.js 安全模式渲染
    ├─ Code → CodeMirror 语法高亮
    ├─ Markdown → react-markdown 渲染
    ├─ Text → 纯文本展示
    ├─ Document → docx-preview / pptx-preview / PDF.js
    ├─ Image → 直接展示 + 缩放
    ├─ Video → HTML5 video player
    └─ LocalService → 本地服务 URL 展示
```

## 七、与 YokoBot (WeChatAutoBot) 的关系

当前工作区还关联了 `C:\Users\10137\Downloads\YokoAIBot\WeChatAutoBot`，这是一个**微信 PC 客户端自动化机器人**项目：

- **YokoBot**: 基于 Windows UI Automation 的微信桌面端自动化，通过 `pywinauto` 操控微信 UI
- **LobsterAI**: 基于 IM 协议/API 层的多平台 Agent 集成，通过 OpenClaw 插件 (`openclaw-weixin`) 对接微信

两者在微信接入层存在技术路线差异，但在 AI Agent 执行、定时任务调度等上层逻辑上存在大量可复用的设计模式。
