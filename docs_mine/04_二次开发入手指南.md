## 一、环境搭建

### 1.1 系统要求

| 依赖 | 版本 | 说明 |
|------|------|------|
| Node.js | **>=24 <25** | 严格版本要求 |
| npm | 内置 | 随 Node.js 安装 |
| Git | — | Windows 需要 PortableGit (详见 README) |
| Python | 3.10+ | 部分 Skill 需要 |

### 1.2 首次启动

```bash
cd e:\code\LobsterAI

# 安装依赖
npm install

# 首次启动：必须使用此命令构建 OpenClaw runtime
# 首次会自动克隆并构建 OpenClaw，可能需要几分钟
npm run electron:dev:openclaw
```

> **重要**: Cowork 会话依赖 OpenClaw runtime。**首次运行必须用 `electron:dev:openclaw`** 构建 runtime。后续日常开发可使用更快的 `npm run electron:dev`，它会复用已构建的 runtime。

### 1.3 常用开发命令

```bash
npm run electron:dev          # 开发模式
npm run electron:dev:openclaw # 带 OpenClaw 构建的开发模式
npm run build                 # 生产构建
npm run lint                  # ESLint 检查
npm test                      # 运行 Vitest 单元测试
npm test -- <filter>          # 运行指定测试
npm run compile:electron      # 仅编译主进程 TypeScript
npm run dist:win              # 打包 Windows 安装包
```

## 二、开发工作流

### 2.1 文件修改 → 效果可见的路径

| 修改区域 | 热重载？ | 说明 |
|---------|---------|------|
| `src/renderer/**` | ✅ | Vite HMR，保存即可见 |
| `src/main/main.ts` | ⚠️ | 需要重启 Electron (`Ctrl+C` → `npm run electron:dev`) |
| `src/main/libs/**` | ⚠️ | 需要重启 Electron |
| `src/main/preload.ts` | ⚠️ | 需要重启 Electron |
| `src/shared/**` | ✅/⚠️ | 渲染进程引用可 HMR，主进程引用需重启 |
| `SKILLs/**` | 🔄 | 运行时热加载（通过 Skill Manager） |

### 2.2 TypeScript 配置

项目使用两套独立的 TypeScript 配置：

| 配置文件 | 作用域 | 模块系统 | 输出目录 |
|---------|--------|---------|---------|
| `tsconfig.json` | 渲染进程 (React) | ES2020 / ESNext modules | — (Vite 处理) |
| `electron-tsconfig.json` | 主进程 (Electron) | CommonJS | `dist-electron/` |

### 2.3 代码规范要点

1. **路径别名**: `@` = `src/renderer/` (仅 Vite 环境)
2. **命名规范**: 组件 `PascalCase.tsx`，函数 `camelCase`，Slice `*Slice.ts`
3. **字符串常量**: 所有 IPC 通道名、状态码必须使用 `as const` 对象集中定义
4. **国际化**: 所有用户可见文本必须通过 `t('key')` 函数，添加新 key 需同时提供中英文
5. **日志**: 主进程使用 `console.log/warn/error`，自动写入日志文件；调试信息用 `console.debug`
6. **样式**: Tailwind CSS 优先，必要时使用 CSS 变量

## 三、核心扩展点

### 3.1 添加新的 IM 平台适配

**难度**: ⭐⭐⭐⭐ | **涉及文件**: 5-8 个

```
步骤:
1. 创建 OpenClaw 插件 (或对接已有协议)
   → 参考 src/main/im/imGatewayManager.ts 中已有平台的注册逻辑

2. 在 GUI 增加配置 UI
   → 修改 src/renderer/components/Settings.tsx (IM 设置 Tab)
   → 修改 src/renderer/services/im.ts (IPC 封装)

3. 在 OpenClaw 配置同步层注册新平台
   → 修改 src/main/libs/openclawConfigSync.ts

4. 添加国际化文本
   → 修改 src/renderer/services/i18n.ts (中英文 key)
   → 修改 src/main/i18n.ts (主进程文本)
```

### 3.2 添加新的 Skill

**难度**: ⭐⭐ | **涉及文件**: 1-3 个

```
步骤:
1. 在 SKILLs/ 目录创建新 Skill 文件夹:
   SKILLs/my-skill/
   ├── SKILL.md           # Skill 入口文件 (YAML frontmatter + Markdown 指令)
   └── scripts/           # 可选: 辅助脚本

   SKILL.md 的 frontmatter 示例:
   ---
   name: my-skill
   description: 我的自定义技能
   version: 1.0.0
   ---
   # 技能指令内容...

2. 修改 SKILLs/skills.config.json 添加启用配置

3. 重启应用或在设置中刷新 Skill 列表
```

### 3.3 添加新的 Artifact 渲染器

**难度**: ⭐⭐⭐ | **涉及文件**: 3-4 个

```
步骤:
1. 创建渲染器组件
   → src/renderer/components/artifacts/renderers/MyRenderer.tsx

2. 注册到 ArtifactRenderer 分发入口
   → 修改 src/renderer/components/artifacts/ArtifactRenderer.tsx
   → 在 switch-case 中添加新类型的渲染分支

3. 更新类型定义 (最先做)
   → 修改 src/renderer/types/artifact.ts
   → 在 ArtifactTypeValue 中添加新类型
   → 在 PREVIEWABLE_ARTIFACT_TYPES 中注册 (如需预览)

4. 更新 ArtifactParser 检测逻辑
   → 修改 src/renderer/services/artifactParser.ts
   → 添加新类型的代码块识别规则

5. 更新预览卡片策略 (可选)
   → 修改 src/renderer/components/artifacts/previewCardPolicy.ts
```

### 3.4 添加新的 LLM Provider

**难度**: ⭐⭐⭐ | **涉及文件**: 3-5 个

```
步骤:
1. 在 API 服务中注册 Provider
   → 修改 src/renderer/services/api.ts
   → 添加 Provider 检测规则 (模型前缀映射)
   → 如果 API 格式特殊，添加新的 chatWith* 方法

2. 在 OpenClaw 配置同步中注册
   → 修改 src/main/libs/openclawConfigSync.ts
   → 将 GUI 配置转换为 OpenClaw Provider 配置

3. 在设置界面添加 Provider 选项
   → 修改 src/renderer/components/Settings.tsx (模型设置 Tab)

4. 在 OpenClaw 模型注入中注册
   → 修改 src/main/libs/openclawAgentModels.ts
```

### 3.5 添加新的 IPC 通道

**难度**: ⭐⭐ | **涉及文件**: 3 个

```
步骤:
1. 定义通道常量
   → 在 src/shared/yourFeature/constants.ts 中:
   export const YourIpcChannel = {
     DoSomething: 'your:doSomething',
   } as const;

2. 主进程注册处理器
   → 在 src/main/main.ts 或 src/main/ipcHandlers/ 中:
   ipcMain.handle(YourIpcChannel.DoSomething, async (_, args) => {
     return result;
   });

3. Preload 桥接暴露
   → 在 src/main/preload.ts 中:
   your: {
     doSomething: (args) => ipcRenderer.invoke(YourIpcChannel.DoSomething, args),
   },
```

### 3.6 添加定时任务策略

**难度**: ⭐⭐⭐⭐ | **涉及文件**: 4-6 个

```
步骤:
1. 在 src/scheduledTask/constants.ts 中添加新的 OriginKind 和 BindingKind

2. 创建新的 Policy 类
   → 参考 src/scheduledTask/policies/ 目录下已有策略

3. 在 PolicyRegistry 中注册
   → 修改策略分发逻辑

4. 在 GUI 增加对应 UI
   → 修改定时任务相关组件
```

## 四、推荐切入方向

按照从简到深的顺序，推荐以下二次开发入手路径：

### 🟢 Level 1: 快速见效 (1-2 天)

**方向 A: 添加自定义 Skill**
- 不需要修改源码，只需在 `SKILLs/` 目录创建文件
- 立即可用，Agent 会自动加载
- 最适合快速扩展 Agent 能力

**方向 B: 修改 UI 文案/样式**
- 修改 `src/renderer/services/i18n.ts` 调整文案
- 修改 `src/renderer/index.css` 或 Tailwind 类调整样式
- Vite HMR 即时可见

### 🟡 Level 2: 功能扩展 (3-5 天)

**方向 C: 添加新的 Artifact 渲染器**
- 理解 Artifact 检测 + 渲染管线
- 创建独立组件，接口清晰
- 可参考已有渲染器（`HtmlRenderer` 最简单）

**方向 D: 扩展 Quick Actions**
- 修改 `src/renderer/services/quickAction.ts`
- 添加首页快捷操作入口
- 可绑定已有 Skill

### 🔴 Level 3: 深度定制 (1-2 周)

**方向 E: 添加新 IM 平台**
- 需要理解 OpenClaw Channel 插件机制
- 需要在 GUI + 主进程 + Gateway 三层同步修改
- 建议先研究 Telegram/Discord 的内置实现

**方向 F: 定制 Agent 行为**
- 修改 `CoworkEngineRouter` 中的事件处理逻辑
- 修改 `OpenClawRuntimeAdapter` 中的 Agent 循环参数
- 需要深入理解 OpenClaw Gateway 的 JSON-RPC 协议

**方向 G: 自定义 Provider 与模型路由**
- 修改 `openclawConfigSync.ts` 的 Provider 映射
- 修改 `openclawAgentModels.ts` 的模型注入逻辑
- 需要理解 OpenClaw 的多 Provider 配置格式

## 五、关键注意事项

### 5.1 ⚠️ 大文件警告

以下文件超过 100KB，修改时要格外小心，建议使用精确的搜索定位而非全文阅读：

| 文件 | 大小 | 建议 |
|------|------|------|
| `main.ts` | 388KB | 搜索 IPC 通道名定位处理器 |
| `openclawRuntimeAdapter.ts` | 345KB | 搜索 WebSocket 事件名定位处理逻辑 |
| `i18n.ts` | 286KB | 搜索 key 名定位翻译文本 |
| `Settings.tsx` | 194KB | 搜索 Tab 名或组件名定位设置项 |
| `CoworkSessionDetail.tsx` | 159KB | 搜索消息类型定位渲染逻辑 |
| `ArtifactPanel.tsx` | 168KB | 搜索 Artifact 类型定位渲染分支 |
| `openclawConfigSync.ts` | 131KB | 搜索 Provider 名定位配置映射 |

### 5.2 ⚠️ 进程隔离边界

```
❌ 渲染进程不能直接 require('fs')
❌ 渲染进程不能直接 require('electron')
❌ 渲染进程不能直接调用 SQLite
✅ 所有系统操作必须通过 IPC → 主进程
✅ 所有 Agent 操作必须通过 Cowork → OpenClaw
```

### 5.3 ⚠️ 修改 Preload 需要同步修改的文件

当你在 `preload.ts` 添加新 API 时，还需要更新 TypeScript 类型声明，使渲染进程能获得类型提示：

```
src/main/preload.ts        ← 添加 API 实现
src/renderer/types/        ← 更新 window.electron 类型声明
```

### 5.4 ⚠️ 测试规范

- 测试文件与源文件同目录，使用 `.test.ts` 扩展名
- 使用 `vitest` 而非 `jest`
- 避免在测试中导入 Electron 专有 API

### 5.5 ⚠️ Git 提交规范

- 使用 Conventional Commits（`feat:`, `fix:`, `chore:` 等）
- 所有 commit message 使用英文
- 不超过 72 字符
- 示例: `feat(cowork): add streaming progress indicator`

## 六、调试技巧

### 6.1 渲染进程调试

- 开发模式下按 `Ctrl+Shift+I` 打开 DevTools
- `console.log` 输出在 DevTools Console 中
- React DevTools 扩展可以检查组件树和 Redux 状态

### 6.2 主进程调试

- 日志自动写入 `%APPDATA%/lobsterai/logs/`
- 使用 `console.log('[Tag] message')` 格式化日志
- 在 `package.json` 的 `electron:dev` 脚本中添加 `--inspect` 开启 Node.js 调试端口

### 6.3 OpenClaw Gateway 调试

- Gateway 日志位于 `%APPDATA%/lobsterai/openclaw/logs/`
- OpenClaw 每日日志位于 `C:/tmp/openclaw/` 或 `%TEMP%/openclaw/`
- 通过 `http://localhost:{port}/` 可以访问 Gateway 的 Web UI（如果已启用）

### 6.4 IPC 通信调试

在 `preload.ts` 中临时添加日志拦截：

```typescript
// 临时调试：记录所有 IPC 调用
const originalInvoke = ipcRenderer.invoke;
ipcRenderer.invoke = (channel, ...args) => {
  console.log(`[IPC] ${channel}`, args);
  return originalInvoke(channel, ...args);
};
```

## 七、架构决策树：我应该改哪里？

```
你要做什么？
│
├─ 改 UI 展示？ → src/renderer/components/
│   ├─ 改 Cowork 会话相关 → components/cowork/
│   ├─ 改 Artifact 渲染 → components/artifacts/
│   ├─ 改 IM 相关 → components/im/
│   ├─ 改定时任务 → components/scheduledTasks/
│   └─ 改设置页面 → components/Settings.tsx
│
├─ 改业务逻辑？ → src/renderer/services/ 或 src/main/
│   ├─ 改 LLM 调用 → services/api.ts
│   ├─ 改会话流程 → services/cowork.ts + main/coworkStore.ts
│   ├─ 改 IM 逻辑 → services/im.ts + main/im/
│   └─ 改 Agent 执行 → main/libs/agentEngine/
│
├─ 改 OpenClaw 行为？ → src/main/libs/openclaw*
│   ├─ 改引擎启动 → openclawEngineManager.ts
│   ├─ 改配置映射 → openclawConfigSync.ts
│   ├─ 改 Agent 参数 → openclawRuntimeAdapter.ts
│   └─ 改记忆系统 → openclawMemoryFile.ts
│
├─ 加新功能？
│   ├─ 新 IPC 通道 → shared/constants.ts + main.ts + preload.ts
│   ├─ 新 Redux 状态 → store/slices/newSlice.ts
│   ├─ 新数据库表 → main/sqliteStore.ts (迁移)
│   └─ 新 Skill → SKILLs/my-skill/
│
└─ 改数据存储？ → src/main/
    ├─ KV 配置 → sqliteStore.ts
    ├─ 会话/消息 → coworkStore.ts
    ├─ IM 配置 → im/imStore.ts
    └─ 定时任务 → scheduledTask/
```

## 八、一句话总结

> **二次开发入门的最佳路径：先创建一个 Skill，理解 Agent 的能力边界；再修改一个 Artifact 渲染器，理解数据流管线；最后再深入 Engine 层定制 Agent 行为。**
