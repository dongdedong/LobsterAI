## 一、进程模型与通信架构

LobsterAI 运行时由**三个独立进程**组成，各进程通过严格的通信协议互联：

```
┌──────────────────────┐     IPC (contextBridge)     ┌──────────────────────────┐
│   渲染进程 (React)     │ ←─────────────────────────→ │    主进程 (Electron)        │
│                      │   window.electron.*          │                          │
│  - CoworkView        │   80+ 通道                   │  - main.ts (10,446 行)    │
│  - Settings          │                              │  - CoworkStore           │
│  - ArtifactPanel     │                              │  - IMGatewayManager      │
│  - Redux Store       │                              │  - SkillManager          │
└──────────────────────┘                              │  - SqliteStore           │
                                                      │  - OpenClawEngineManager │
                                                      └──────────┬───────────────┘
                                                                 │ WebSocket (JSON-RPC)
                                                                 │ ws://127.0.0.1:{port}
                                                      ┌──────────▼───────────────┐
                                                      │  OpenClaw Gateway (子进程)  │
                                                      │                          │
                                                      │  - Agent 执行循环          │
                                                      │  - Cron 调度引擎          │
                                                      │  - Channel Adapters      │
                                                      │  - Tool Registry         │
                                                      └──────────────────────────┘
```

### 1.2 主进程 → OpenClaw Gateway 通信

主进程通过 `OpenClawRuntimeAdapter` (344KB) 建立与 Gateway 的 WebSocket 长连接：

```typescript
// 核心通信模式
gatewayClient.request('chat.send', { sessionKey, prompt, ... })    // → 发起 Agent 对话
gatewayClient.request('cron.add', { schedule, payload, ... })       // → 创建定时任务
gatewayClient.request('channels.status', { probe: false })          // → 查询 IM 通道状态
gatewayClient.request('sessions.list', { ... })                     // → 列出 OpenClaw 会话
```

Gateway 通过 WebSocket 推送以下事件：
- `chat.delta` / `chat.final` → 流式 Agent 回复
- `exec.approval.requested` → 工具调用需要用户审批
- `session.status` → 会话状态变更

## 二、Cowork 引擎路由机制

`CoworkEngineRouter` 是主进程中最关键的设计模式之一。它实现了 `CoworkRuntime` 接口，将所有会话操作统一路由到 OpenClaw 引擎：

```typescript
// src/main/libs/agentEngine/coworkEngineRouter.ts
export class CoworkEngineRouter extends EventEmitter implements CoworkRuntime {
  private readonly runtime: CoworkRuntime;  // OpenClaw 适配器
  private readonly sessionEngine = new Map<string, CoworkAgentEngine>();  // 每个会话绑定的引擎
  private readonly requestEngine = new Map<string, CoworkAgentEngine>();  // 每个权限请求绑定的引擎

  async startSession(sessionId, prompt, options) {
    const engine = this.safeResolveEngine();  // 解析当前引擎
    this.sessionEngine.set(sessionId, engine);
    await this.runtime.startSession(sessionId, prompt, options);
  }
}
```

**设计价值**：
- GUI 永远消费标准化的 Cowork 事件（`message` / `messageUpdate` / `permissionRequest` / `complete`），不感知底层引擎
- 切换引擎时自动清理所有活动会话，避免跨引擎上下文污染
- 权限请求通过 `requestEngine` Map 正确路由回发起请求的引擎

## 四、OpenClaw 引擎生命周期

`OpenClawEngineManager` 管理 Gateway 子进程的完整生命周期：

```
not_installed ──→ ready ──→ starting ──→ running
      │                         │           │
      │                         ▼           ▼
      │                       error ←────── │ (崩溃/超时)
      │                         │           │
      └─────────────────────────┘           │
               (重试)                       │ (自动重启, 最多 5 次)
                                            │ 退避: 3s → 5s → 10s → 20s → 30s
```

### 关键实现细节

1. **端口分配**：从 18789 开始扫描可用端口（最多扫描 80 个）
2. **Token 认证**：每次启动生成 SHA-256 随机 Token，通过环境变量注入 Gateway
3. **Windows 优化**：在 Windows 上使用 `child_process.spawn` + `ELECTRON_RUN_AS_NODE=1` 替代 `utilityProcess.fork`，减少 5 倍冷启动开销
4. **健康检查**：启动后通过 HTTP 探针（`/health`）等待 Gateway 就绪，超时 300 秒
5. **编译缓存**：通过 `NODE_COMPILE_CACHE` 加速 ESM 模块加载
6. **V8 内存限制**：设置 `--max-old-space-size=4096` 防止 OOM

### 6.1 API 格式分发

`ApiService.chat()` 根据 Provider 和 API 格式分发到三个独立的调用路径：

```typescript
// src/renderer/services/api.ts
async chat(message, onProgress, history) {
  const provider = this.detectProvider(modelId, providerHint);
  const apiFormat = this.normalizeApiFormat(config.apiFormat);

  switch (apiFormat) {
    case 'gemini':    return this.chatWithGemini(...);     // Gemini 原生协议
    case 'anthropic': return this.chatWithAnthropic(...);  // Anthropic /v1/messages
    default:          return this.chatWithOpenAICompatible(...); // OpenAI /v1/chat/completions
  }
}

### 7.2 消息处理统一入口

所有平台的入站消息都汇聚到同一个 `messageHandler`：

```typescript
const messageHandler = async (message: IMMessage, replyFn) => {
  // 1. 持久化通知目标 (用于后续主动推送)
  this.persistNotificationTarget(message.platform);

  // 2. 优先使用 Cowork 模式 (完整 Agent 能力)
  if (this.coworkHandler) {
    response = await this.coworkHandler.processMessage(message);
  } else {
    // 3. 降级到简单聊天模式
    response = await this.chatHandler.processMessage(message);
  }

  await replyFn(response);
};
```

## 十、认证与安全

### 10.1 OAuth 双 Token 滚动刷新

```
                    ┌───────────────────────────────┐
                    │     Portal 登录页 (浏览器)      │
                    │     URS 身份验证               │
                    └───────────┬───────────────────┘
                                │ deep link
                                │ lobsterai://auth/callback?code=xxx
                                ▼
                    ┌───────────────────────────────┐
                    │     主进程 AuthCallbackRouter    │
                    │     POST /api/auth/exchange    │
                    │     → accessToken (2h)         │
                    │     → refreshToken (30d)       │
                    │     → SQLite auth_tokens       │
                    └───────────────────────────────┘

正常请求:
  fetchWithAuth() → Authorization: Bearer {accessToken}

被动刷新 (HTTP 401):
  → POST /api/auth/refresh (refreshToken)
  → 新 accessToken + 新 refreshToken (滚动续期)
  → 重试原请求

主动刷新:
  定时检查 → exp < 5 分钟 → 后台静默刷新

退出条件:
  连续 30 天不使用 → refreshToken 过期 → 重新登录
```

### 10.2 OpenClaw Gateway 安全

- Gateway 仅绑定 `127.0.0.1`（loopback），不暴露外网
- 通过随机 Token 认证 WebSocket 连接
- 敏感凭证通过环境变量注入，不写入配置文件
- Agent 工具调用需要用户审批
